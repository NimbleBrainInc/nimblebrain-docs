The NimbleBrain Helm chart deploys two components: a **platform** StatefulSet (agent engine + API) and a **web** Deployment (React UI), with an optional Ingress for external access.

## Prerequisites

- Kubernetes 1.26+
- Helm 3.12+
- `kubectl` configured for your cluster
- An [Anthropic API key](https://console.anthropic.com/)

## Install

<Steps>

1. **Clone the repository**

   The Helm chart lives in the NimbleBrain source tree:

   ```bash
   git clone https://github.com/NimbleBrainInc/nimblebrain.git
   cd nimblebrain
   ```

2. **Preview the rendered manifests**

   ```bash
   helm template nimblebrain ./helm \
     --set secrets.anthropicApiKey=sk-ant-api03-placeholder \
     --set secrets.nbApiKey=placeholder-key-32chars-minimum
   ```

   Review the output to confirm resources match your cluster expectations.

3. **Install the release**

   ```bash
   helm install nimblebrain ./helm \
     --set secrets.anthropicApiKey=sk-ant-api03-your-key-here \
     --set secrets.nbApiKey=change-me-to-a-strong-secret-at-least-32-chars \
     --set ingress.hostname=nb.example.com
   ```

   ```
   NAME: nimblebrain
   STATUS: deployed
   REVISION: 1
   ```

4. **Verify the pods**

   ```bash
   kubectl get pods -l app.kubernetes.io/instance=nimblebrain
   ```

   ```
   NAME                              READY   STATUS    RESTARTS   AGE
   nimblebrain-platform-0            1/1     Running   0          45s
   nimblebrain-web-6d4f8b7c9-xk2lm  1/1     Running   0          45s
   ```

</Steps>

## Architecture

The chart creates these Kubernetes resources:

| Resource | Name suffix | Description |
|----------|-------------|-------------|
| StatefulSet | `-platform` | Single-replica agent engine with persistent storage |
| Deployment | `-web` | Scalable web UI (Caddy + React SPA) |
| Service | `-platform` | ClusterIP on port 3000 |
| Service | `-web` | ClusterIP on port 8080 |
| Secret | (release name) | Anthropic API key and NB API key |
| ConfigMap | (release name) | `nimblebrain.json` configuration |
| Ingress | (release name) | Routes `/v1/*` to platform, `/` to web |
| PVC | `data` (via VCT) | Persistent workspace volume for platform |

The Ingress routes traffic based on path:
- `/v1/*` goes to the **platform** service (port 3000) for API requests
- `/` goes to the **web** service (port 8080) for the UI

## values.yaml reference

### Platform

```yaml
platform:
  image:
    repository: nimblebrain/platform
    tag: latest
    pullPolicy: IfNotPresent
  resources:
    requests:
      cpu: 500m
      memory: 512Mi
    limits:
      cpu: 2000m
      memory: 2Gi
  persistence:
    enabled: true
    size: 10Gi
    storageClass: ""
  env:
    NB_MAX_ITERATIONS: "10"
```

| Value | Type | Default | Description |
|-------|------|---------|-------------|
| `platform.image.repository` | string | `nimblebrain/platform` | Container image registry and name |
| `platform.image.tag` | string | `latest` | Image tag |
| `platform.image.pullPolicy` | string | `IfNotPresent` | Kubernetes pull policy |
| `platform.resources.requests.cpu` | string | `500m` | CPU request |
| `platform.resources.requests.memory` | string | `512Mi` | Memory request |
| `platform.resources.limits.cpu` | string | `2000m` | CPU limit |
| `platform.resources.limits.memory` | string | `2Gi` | Memory limit |
| `platform.persistence.enabled` | bool | `true` | Create a PersistentVolumeClaim for `/data` |
| `platform.persistence.size` | string | `10Gi` | PVC size |
| `platform.persistence.storageClass` | string | `""` | Storage class name. Empty string uses cluster default |
| `platform.env` | map | `{NB_MAX_ITERATIONS: "10"}` | Additional environment variables injected into the platform container |

The platform StatefulSet always has a single replica. It mounts the ConfigMap as `/app/nimblebrain.json` (read-only) and the PVC as `/data`. The environment variables `ANTHROPIC_API_KEY`, `NB_API_KEY`, `NB_WORK_DIR`, and `NB_HOST` are set automatically from secrets and chart defaults.

### Web

```yaml
web:
  image:
    repository: nimblebrain/web
    tag: latest
    pullPolicy: IfNotPresent
  replicas: 1
  resources:
    requests:
      cpu: 100m
      memory: 64Mi
    limits:
      cpu: 500m
      memory: 256Mi
```

| Value | Type | Default | Description |
|-------|------|---------|-------------|
| `web.image.repository` | string | `nimblebrain/web` | Container image |
| `web.image.tag` | string | `latest` | Image tag |
| `web.image.pullPolicy` | string | `IfNotPresent` | Pull policy |
| `web.replicas` | int | `1` | Number of web pods |
| `web.resources.requests.cpu` | string | `100m` | CPU request |
| `web.resources.requests.memory` | string | `64Mi` | Memory request |
| `web.resources.limits.cpu` | string | `500m` | CPU limit |
| `web.resources.limits.memory` | string | `256Mi` | Memory limit |

The web Deployment is stateless and can scale horizontally. Each pod runs Caddy, which serves the built React SPA and reverse-proxies `/v1/*` to the platform Service.

### Ingress

```yaml
ingress:
  enabled: true
  hostname: nb.example.com
  tls: true
  annotations: {}
```

| Value | Type | Default | Description |
|-------|------|---------|-------------|
| `ingress.enabled` | bool | `true` | Create an Ingress resource |
| `ingress.hostname` | string | `nb.example.com` | Hostname for the Ingress rules |
| `ingress.tls` | bool | `true` | Enable TLS. Creates a TLS secret reference named `<release>-nimblebrain-tls` |
| `ingress.annotations` | map | `{}` | Annotations for the Ingress (e.g., cert-manager, ingress class) |

When TLS is enabled, the Ingress references a secret named `<release>-nimblebrain-tls`. You must provision this secret yourself, either manually or with cert-manager.

### Secrets

```yaml
secrets:
  anthropicApiKey: ""
  nbApiKey: ""
```

| Value | Type | Default | Description |
|-------|------|---------|-------------|
| `secrets.anthropicApiKey` | string | `""` | Anthropic API key for Claude |
| `secrets.nbApiKey` | string | `""` | API authentication key (minimum 8 characters) |

Both values are base64-encoded and stored in a Kubernetes Secret. The platform container reads them as environment variables via `secretKeyRef`.

<Aside type="danger">
Do not commit secrets to version control. Pass them via `--set` on the command line, or use an external secret manager (e.g., Sealed Secrets, External Secrets Operator, Vault) and reference the secret by name.
</Aside>

### Configuration

```yaml
config:
  nimblebrainJson: |
    {
      "bundles": [],
      "skillDirs": ["/data/skills"]
    }
```

| Value | Type | Default | Description |
|-------|------|---------|-------------|
| `config.nimblebrainJson` | string | (see above) | The full contents of `nimblebrain.json`, rendered into a ConfigMap |

To pre-install bundles, add them to the `bundles` array:

```yaml
config:
  nimblebrainJson: |
    {
      "bundles": [
        { "name": "@nimblebraininc/bash" },
        { "name": "@nimblebraininc/ipinfo" }
      ],
      "skillDirs": ["/data/skills"]
    }
```

## Health checks

The platform StatefulSet configures both liveness and readiness probes:

```yaml
livenessProbe:
  httpGet:
    path: /v1/health
    port: http
  initialDelaySeconds: 10
  periodSeconds: 30
  timeoutSeconds: 5
readinessProbe:
  httpGet:
    path: /v1/health
    port: http
  initialDelaySeconds: 5
  periodSeconds: 10
  timeoutSeconds: 5
```

The web Deployment probes on `/` (the index page):

```yaml
livenessProbe:
  httpGet:
    path: /
    port: http
  initialDelaySeconds: 5
  periodSeconds: 30
  timeoutSeconds: 5
readinessProbe:
  httpGet:
    path: /
    port: http
  initialDelaySeconds: 3
  periodSeconds: 10
  timeoutSeconds: 5
```

## Ingress with cert-manager

To automatically provision TLS certificates using cert-manager:

```bash
helm install nimblebrain ./helm \
  --set secrets.anthropicApiKey=sk-ant-api03-your-key-here \
  --set secrets.nbApiKey=change-me-to-a-strong-secret-at-least-32-chars \
  --set ingress.hostname=nb.example.com \
  --set ingress.tls=true \
  --set ingress.annotations."cert-manager\.io/cluster-issuer"=letsencrypt-prod
```

This tells cert-manager to issue a certificate for `nb.example.com` and store it in the `<release>-nimblebrain-tls` secret that the Ingress references.

## Custom values file

For repeatable deployments, put your overrides in a file:

```yaml title="values-production.yaml"
platform:
  image:
    tag: "0.1.0"
  resources:
    requests:
      cpu: 1000m
      memory: 1Gi
    limits:
      cpu: 4000m
      memory: 4Gi
  persistence:
    size: 50Gi
    storageClass: gp3
  env:
    NB_MAX_ITERATIONS: "15"
    ALLOWED_ORIGINS: "https://nb.example.com"

web:
  replicas: 2
  image:
    tag: "0.1.0"

ingress:
  hostname: nb.example.com
  tls: true
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
    kubernetes.io/ingress.class: nginx
```

```bash
helm install nimblebrain ./helm \
  -f values-production.yaml \
  --set secrets.anthropicApiKey=sk-ant-api03-your-key-here \
  --set secrets.nbApiKey=change-me-to-a-strong-secret-at-least-32-chars
```

## Upgrade

```bash
helm upgrade nimblebrain ./helm \
  -f values-production.yaml \
  --set secrets.anthropicApiKey=sk-ant-api03-your-key-here \
  --set secrets.nbApiKey=change-me-to-a-strong-secret-at-least-32-chars
```

The platform StatefulSet performs a rolling update. The PVC is retained across upgrades, so workspace data is preserved.

To check the release status:

```bash
helm status nimblebrain
```

## Uninstall

```bash
helm uninstall nimblebrain
```

<Aside type="caution">
`helm uninstall` does **not** delete PersistentVolumeClaims. Your workspace data in the `data-nimblebrain-platform-0` PVC is preserved. Delete it manually if you want to remove all data:

```bash
kubectl delete pvc data-nimblebrain-platform-0
```
</Aside>

## Scaling considerations

The **platform** StatefulSet runs a single replica. It manages MCP bundle subprocesses and in-memory state, so horizontal scaling is not supported. Scale vertically by increasing `platform.resources.limits`.

The **web** Deployment is stateless. Increase `web.replicas` to handle more concurrent UI connections. Each web pod runs Caddy and proxies API calls to the single platform pod.

For high availability of the agent engine itself, run multiple NimbleBrain installations in separate namespaces with a load balancer in front.
