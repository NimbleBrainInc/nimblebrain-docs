`nb credential` manages workspace-scoped secrets: OAuth `client_secret` values, API keys, and other opaque strings that a workspace's bundles need at runtime. It is the command-line way to seed the secrets that a `workspace.json` references through `{ ref: "credential", key: "..." }`.

The command is also available under the shorter alias `nb creds`.

<Aside type="note">
  Credentials are scoped to a **workspace**, not to a user or to the whole instance. Every command takes a workspace id (`<wsId>`) as its first argument.
</Aside>

## Usage

```
nb credential <action> <wsId> [args] [flags]
nb creds <action> <wsId> [args] [flags]
```

## Actions

| Action | Description |
|--------|-------------|
| `set <wsId> <key> <value>` | Set or replace a secret value |
| `get <wsId> <key>` | Print a secret value to stdout (designed for piping) |
| `delete <wsId> <key>` (alias `rm`) | Remove a secret (idempotent) |
| `list <wsId>` (alias `ls`) | List the secret keys for a workspace (values are not printed) |

## Flags

| Flag | Description | Default |
|------|-------------|---------|
| `--work-dir <path>` | Override the default work directory | `$NB_WORK_DIR`, else `~/.nimblebrain` |

## Setting a secret

```bash
nb credential set ws_1a2b3c4d5e6f7a8b hubspot.client_secret sk-abc123
```

```
[credential] set ws_1a2b3c4d5e6f7a8b:hubspot.client_secret (9 chars)
```

The value is written atomically. Setting an existing key replaces its value. Keys follow the same shape as bundle config keys: dotted namespaces with alphanumerics, hyphens, and underscores (for example `google.oauth-client`). Slashes and `..` are rejected.

<Aside type="note">
  `set` reports the **length** of the value, never the value itself. The secret is not echoed back.
</Aside>

## Getting a secret

`get` writes the raw secret to stdout with no log decoration and no trailing newline, so it pipes cleanly into another command:

```bash
curl -H "Authorization: Bearer $(nb credential get ws_1a2b3c4d5e6f7a8b api.token)" \
  https://api.example.com/v1/me
```

If the key is not set, `get` writes a message to stderr and exits non-zero:

```bash
nb credential get ws_1a2b3c4d5e6f7a8b missing.key
```

```
[credential] not found: ws_1a2b3c4d5e6f7a8b:missing.key
```

## Listing keys

`list` (alias `ls`) prints the keys stored for a workspace, one per line. Values are never printed:

```bash
nb credential list ws_1a2b3c4d5e6f7a8b
```

```
api.token
hubspot.client_secret
```

When the workspace has no secrets:

```bash
nb creds ls ws_1a2b3c4d5e6f7a8b
```

```
[credential] no secrets in ws_1a2b3c4d5e6f7a8b
```

## Deleting a secret

`delete` (alias `rm`) removes a secret. It is idempotent: deleting a key that does not exist succeeds silently.

```bash
nb credential delete ws_1a2b3c4d5e6f7a8b hubspot.client_secret
```

```
[credential] deleted ws_1a2b3c4d5e6f7a8b:hubspot.client_secret
```

## Where credentials are stored

Secrets live on disk under the workspace, one file per key:

```
{workDir}/workspaces/{wsId}/credentials/secrets/{key}
```

Each file is written with mode `0600` (owner read/write only). The default `{workDir}` is `~/.nimblebrain`, overridable per command with `--work-dir` or globally with the `NB_WORK_DIR` environment variable. Pointing `--work-dir` at a non-default deployment is useful when seeding secrets from a one-off shell.

<Aside type="note">
  The self-hosted runtime stores these values as plaintext files protected by filesystem permissions. The store is an interface — a managed deployment can swap in an encrypted backend without changing these commands.
</Aside>
