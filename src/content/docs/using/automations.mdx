Automations are scheduled agent runs. Instead of you typing a prompt, NimbleBrain runs a saved prompt on a schedule — a daily digest, a recurring report, a periodic check — and the agent executes it unattended, with access to your tools and connectors.

Each automation is a prompt plus a schedule. When it fires, the runtime starts an agent run framed around producing a deliverable rather than a back-and-forth conversation.

## What an automation is

An automation has:

- **A name** — becomes a kebab-case ID (for example, "Weekly Report" → `weekly-report`)
- **A prompt** — the instruction sent to the agent on each run
- **A schedule** — either a cron expression or a fixed interval
- **Optional run-time policy** — a model override, a forced skill match, iteration and token caps

## Who an automation runs as

Automations are **owned by the user who creates them**, not by a workspace. A scheduled run fires as its owner, focused on the workspace it was created in, but with full reach across every workspace that owner belongs to.

This has one practical consequence: if an automation needs to reach a connector (for example, to post to Teams or Slack), that connector must be installed in a **shared** workspace the owner is a member of. A connector that lives only in someone's personal workspace is never reachable by another user's automation.

<Aside type="note">
Automation runs use your **timezone** for cron scheduling. The default is `Pacific/Honolulu`, overridable by the operator with the `NB_TIMEZONE` environment variable. A schedule can also carry its own IANA timezone.
</Aside>

## Schedules

Two schedule types:

- **Cron** — a standard 5-field cron expression (for example, `0 9 * * 1` for 9:00 AM every Monday)
- **Interval** — a fixed interval in milliseconds, minimum 60000 (one minute)

## Creating an automation

The simplest way is to ask the agent in chat. Describe what you want and when:

```
Every weekday at 8am, summarize my unread email and post it to the
#daily channel.
```

The agent creates the automation for you, generating the schedule and the prompt. You can ask it to list, pause, resume, or delete automations the same way.

You can also browse and manage automations from the **Automations** panel in the sidebar.

## Managing automations from the CLI

The `nb automation` command manages automations from the terminal. Every subcommand accepts a global `--json` flag for machine-readable output.

<Steps>

1. **List automations:**

   ```bash
   nb automation list
   ```

   Shows a table of name, enabled state, schedule, source, last run status, and next run time.

2. **Inspect one automation:**

   ```bash
   nb automation status weekly-report
   ```

   Prints the full definition (schedule, model, caps, prompt preview) and recent run history with per-run status, start time, iterations, and tool-call counts.

3. **Pause an automation** (disable it without deleting):

   ```bash
   nb automation pause weekly-report
   ```

4. **Resume a paused automation:**

   ```bash
   nb automation resume weekly-report
   ```

5. **Trigger an immediate run**, bypassing the schedule:

   ```bash
   nb automation run weekly-report
   ```

   Short runs return the full run record. Runs that take longer than about 30 seconds return a `dispatched` acknowledgement and continue in the background — poll `nb automation status <name>` to watch for completion.

</Steps>

<Aside type="tip">
Pausing keeps the definition and its run history intact; the scheduler simply skips it. Resume re-enables it on its existing schedule. To remove an automation entirely, ask the agent to delete it (run history is preserved even then).
</Aside>

## Run history

Every run is recorded with its status, start and completion times, iteration count, tool calls, and any error. View it through `nb automation status <name>`, the Automations panel, or by asking the agent. If the runtime restarts while a run is in flight, the orphaned run is swept and marked failed so history stays accurate.
